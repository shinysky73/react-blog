# PRD: 블로그 게시물 생성 기능

## 1. 개요

**목표:** 인증된 사용자가 애플리케이션 내에서 새로운 블로그 게시물을 작성하고 게시할 수 있는 기능을 제공합니다.

## 2. 대상 사용자

이 애플리케이션에 로그인한 모든 인증된 사용자.

## 3. 사용자 스토리

- **사용자로서,** 로그인한 상태에서 내 생각을 공유하기 위해 제목과 내용이 있는 새 블로그 게시물을 작성하고 싶습니다.
- **사용자로서,** 게시물을 작성할 때 양식을 잘못 제출하면(예: 제목 없음) 명확한 유효성 검사 오류를 보고 싶습니다.
- **사용자로서,** 게시물을 성공적으로 생성한 후에는 대시보드나 확인 페이지로 리디렉션되어 성공적으로 완료되었음을 확인하고 싶습니다.

## 4. 기능 요구사항

### FR1: 게시물 생성 양식
- 사용자가 새 게시물을 작성할 수 있는 전용 UI 페이지가 있어야 합니다.
- 양식에는 '제목'과 '내용'을 위한 입력 필드가 포함되어야 합니다.

### FR2: 양식 유효성 검사
- **기술 스택:** 프런트엔드와 백엔드 모두에서 `zod`를 사용하여 유효성 검사 스키마를 관리합니다.
- **필드 규칙:**
    - `제목`: 필수 필드이며 비워 둘 수 없습니다.
    - `내용`: 선택적 필드입니다.
- **피드백:**
    - 클라이언트 측 유효성 검사는 제출 즉시 사용자에게 명확한 오류 메시지를 제공해야 합니다.
    - 서버 측 유효성 검사도 구현되어 API를 직접 호출하는 경우를 방지해야 합니다.

### FR3: API 엔드포인트
- 게시물 생성을 처리하기 위한 백엔드 API 엔드포인트 `POST /api/posts`가 필요합니다.
- 이 엔드포인트는 `title` (문자열)과 `content` (선택적 문자열)를 포함하는 JSON 페이로드를 수락해야 합니다.

### FR4: 인증 및 권한 부여
- 오직 인증된(로그인한) 사용자만이 새 게시물을 생성할 수 있습니다.
- `POST /api/posts` 엔드포인트는 JWT 인증을 통해 보호되어야 합니다.
- 게시물이 생성될 때, 게시물의 작성자는 현재 로그인한 사용자와 자동으로 연결되어야 합니다.

### FR5: 사용자 피드백
- 게시물 생성이 성공하거나 실패했을 때 사용자에게 명확한 피드백(예: 토스트 알림)을 제공해야 합니다.

### FR6: 기능 접근성
- 사용자가 게시물 생성 페이지로 쉽게 이동할 수 있는 명확한 진입점(예: 대시보드의 '게시물 작성' 버튼)이 있어야 합니다.

## 5. 비기능 요구사항

### NFR1: 보안
- 게시물 생성 엔드포인트는 CSRF 및 기타 잠재적 공격으로부터 보호되어야 합니다.
- 한 사용자가 다른 사용자를 대신하여 게시물을 생성할 수 없도록 해야 합니다.

### NFR2: 성능
- 양식 제출 및 게시물 생성 프로세스는 응답성이 뛰어나야 하며, 일반적인 네트워크 조건에서 500ms 이내에 완료되어야 합니다.

## 6. 범위 외 (Out of Scope)

- 기존 게시물 수정 또는 삭제 기능.
- 게시물 목록 또는 단일 게시물 조회 기능.
- 게시물에 이미지나 다른 미디어 파일 첨부 기능.
- 게시물을 임시 저장하는 기능.
- 비로그인 사용자를 위한 공개 게시물 보기.

## 7. 성공 지표

- 사용자당 생성된 게시물 수.
- 게시물 생성 API 엔드포인트의 낮은 오류율.
- 게시물 생성 양식의 높은 성공적인 제출 완료율.
